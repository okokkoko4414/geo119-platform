## 这个产品到底是什么

Phase A 证明了 GEO119 的技术可行性。Phase B 回答一个更大的问题：**GEO119 能否成为一个真正的全球产品？**

25 种语言是"能工作"。70 种语言是"为世界而建"。差距不在数量——在于当越南用户打开 GEO119 时，产品感觉像是越南团队做的，而不是中国团队翻译的。

这就是 Phase B 的产品方向：**从"可翻译"到"为每个市场而建"。**

---

## 10-Star 版本长什么样

不是"70 种语言都能跑"——那只是 1-star。

10-star 是：**一个巴西用户用 GEO119 三周后，向朋友推荐时说"这个产品很懂我们"。**

差距在哪：
- 1-star: 70 种语言的翻译管道跑通了（机械完成）
- 3-star: 每种语言的翻译质量达到阈值，用户不会因为翻译质量问题流失
- 5-star: 效果追踪显示，新语言市场的用户行为（留存、转化）与成熟市场无统计显著差异
- 10-star: 产品在 70 个市场各自形成口碑循环——用户觉得这就是为他们做的

本次 Phase B 目标是 **3-star**：覆盖 70 语言，每种达到质量阈值，基础设施支撑全量运行。

## 模式选择：Hold Scope，两阶段执行

**不扩展。** CTO V2 计划已获 Board 批准，范围边界清晰。CEO 的价值不在重新定义范围，而在正确的执行顺序和产品判断。

**也不缩减。** 概念验证门已证明委派链能跑通。现在是执行时刻。

**两阶段策略**——这是 CEO 的核心判断：

### Phase B-Core（先构建，上线，拿数据）
B5 基础设施 → B4 English UI → B1 语言扩展 25→70 → B2 基础追踪

**为什么这个顺序？** 基础设施和 UI 是所有东西的前提。语言扩展是核心价值。基础追踪（曝光/点击）足够衡量 B1 是否成功——不需要完整分析套件。

### Phase B-Growth（数据驱动，后构建）
B2 完整效果追踪 → B3 批量优化引擎

**为什么后置？** B2 的转化/留存/多维下钻功能和 B3 的优化引擎是数据饥渴型模块。在 B1 没有真实流量数据之前构建它们，就是在猜。等 B-Core 上线跑了数据，B2 和 B3 的构建会被真实需求驱动，而不是假设。

## 模块级产品方向

### B1 语言扩展 25→70

**产品判断**：不是一次性开 70 种语言。是渐进式管道——先开高资源语言拿质量基线，再扩展到中资源语言。

**质量策略**：
- 高资源语言（已有充足训练数据）：全自动翻译 + 质量自动检测
- 中资源语言（45 个新增）：翻译 + 关键术语人工校验清单 + 回退到高资源语言
- 低资源语言：机器翻译 + 英语回退标注

**验收不止于"70 种语言能加载"。产品验收标准**：随机抽查 10 个中资源语言，找 3 个目标市场母语者做盲评，"能理解"率 > 90%。

### B2 Effect Tracking 效果追踪

**产品判断**：这不是运维看板——这是产品决策引擎。

第一版（B-Core）只需要回答一个问题：**"新语言市场的用户在用产品吗？"** 指标：曝光、点击。够简单，够重要。

第二版（B-Growth）回答：**"他们留下来了吗？为什么？"** 指标：转化、留存、多维下钻（语言 × 地区 × 设备 × 时间）。导出 API 支持外部分析。

### B3 Batch Optimization 批量优化引擎

**产品判断**：这个模块的存在意义是让 $400/月的预算跑出 $4000/月的效果。

如果批量优化引擎自己的运行成本超过它节省的成本，它就是负资产。关键约束：
- 吞吐 ≥10k/小时是硬指标
- 成本 < $0.001/词是硬约束
- P99 < 30s 是用户体验底线
- 但真正的产品成功指标是：**优化引擎运行后的单位成本是否显著低于优化前**

### B4 English UI

**产品判断**：这不是翻译项目。这是品牌项目。

"无硬编码中文"是地板，不是天花板。产品目标：
- 英文 UI 读起来像原生英文产品，不是翻译的
- SEO 就绪（URL 结构、meta 标签、结构化数据）
- 响应式——移动端不是桌面端的缩小版
- WordPress + Blade + Tailwind 保持，不引入新前端框架

### B5 基础设施

**产品判断**：基础设施的产品价值 = 让其他四个模块的团队感觉不到它的存在。

CI/CD 通、监控全、回滚 < 5min——这些都是 baseline。真正的产品标准：**从代码提交到生产部署，开发者只需要 push，其他全部自动化。**
